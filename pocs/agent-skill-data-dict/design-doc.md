# Design Doc — Data Dictionary Skill (`/data-dict`)

## 1. Overview

`/data-dict` is a Claude Code skill that reads a codebase, discovers the **database schema wherever it is defined**, and renders a self-contained light-theme website that documents it. A real project rarely keeps its schema in one place: some tables live in Liquibase changelogs, some are created by raw SQL run through JDBC, and some are generated by Hibernate from JPA entities. This skill reads all three, merges them into one model of the database, and produces a data dictionary anyone on the team can read.

The generated site has three tabs:

1. **Tables** — every table with its columns, types, sizes, keys, a plain-language description, and a badge saying where it came from (Liquibase, SQL/JDBC, or Hibernate).
2. **ER Diagram** — a hand-drawn (Excalidraw-style) entity-relationship diagram of the tables and their relationships.
3. **Query** — an in-browser SQL console that runs against the discovered schema, so anyone can try queries without a live database.

The skill is **generic**. It is not tied to the bundled sample app: it scans whatever repository it is pointed at and discovers whatever schema definitions exist there.

## 2. Problem Statement

The schema is the contract every part of a system depends on, yet on a real codebase it is the hardest thing to see whole. It is scattered:

- **Liquibase / Flyway** changelogs define some tables declaratively.
- **Raw SQL through JDBC** (`jdbcTemplate.execute("CREATE TABLE ...")`, `schema.sql`, init scripts) defines others imperatively.
- **Hibernate / JPA** generates the rest from `@Entity` classes at runtime, so the DDL never appears in source at all.

New engineers cannot answer "what tables exist and what do they mean?" without spinning up the app, dumping the database, and reverse-engineering names. There is no single, readable, current picture of the data model — and the data model is exactly the thing everyone needs to understand first.

This skill produces that picture directly from the source, with no running database required.

## 3. Goals

- Discover tables and columns from **three independent sources** — Liquibase changelogs, raw SQL/JDBC, and JPA/Hibernate entities — and **merge** them into one schema, even when a single database mixes all three.
- Record, per table and per column, **where the definition came from** so the dictionary is auditable.
- Capture column **type, size (length/precision), nullability, primary keys, defaults, and remarks** when the source provides them.
- Detect **relationships** from explicit foreign keys and, where none exist, infer them from `<name>_id → <name>` naming conventions (common when tables span tools).
- Render a **light-theme, self-contained website** with the three required tabs.
- Provide a working **in-browser query console** over the discovered schema.
- Be **generic** — zero assumptions about the sample app; works on any repo.
- Run with **zero third-party Python packages** (stdlib only), like the other skills in this collection.

## 4. Non-Goals

- Not a live database client. The query console runs against an in-browser SQLite built from the discovered DDL, not the project's real database. It is for exploring the schema, not production data.
- Not a migration tool. It reads schema definitions; it never writes, runs, or alters migrations.
- Not a full SQL dialect translator. It normalizes common types to SQLite for the in-browser console; exotic vendor types degrade gracefully to `TEXT`.
- Not a data profiler. With no live database there are no real row counts; "size" means schema size (column count, declared column lengths, estimated row width), never invented row statistics.
- v1 parses **Liquibase** (XML, formatted-SQL, JSON) and **JPA/Hibernate** Java annotations. Flyway is just SQL files, so it is covered by the SQL parser. YAML Liquibase is best-effort.

## 5. Architecture

### 5.1 Pipeline

```
scan repo            parse each source                merge                    render
+--------------+    +--------------------------+    +----------------+    +------------------+
| walk files,  |--->| liquibase changelogs     |--->| unify tables   |--->| inject JSON into |
| classify by  |    | sql / jdbc CREATE TABLE  |    | resolve FKs    |    | template.html    |
| kind         |    | jpa @Entity annotations  |    | est row width  |    | -> index.html    |
+--------------+    +--------------------------+    | sqlite DDL     |    +------------------+
                                                     +----------------+
```

Two components, mirroring the other skills in this collection:

| Component | File | Responsibility |
|---|---|---|
| Engine | `scripts/datadict.py` | Pure Python stdlib. Walks the repo, parses the three schema sources, merges them, resolves relationships, computes deterministic per-table facts, emits `data.json`, and injects it into the template to produce `index.html`. |
| View | `assets/template.html` | Self-contained light-theme page. Inline CSS + vanilla JS. Renders the three tabs. The query tab loads `sql.js` (SQLite compiled to WebAssembly) to run user SQL in the browser. |

The skill (`SKILL.md`) only orchestrates: pick the target directory, run the engine by its installed absolute path, open the report, and summarize what was found. **The schema in the report is parsed from source, not invented by the model.**

### 5.2 Why a deterministic engine, not the model

Reading a schema "by eye" with the model would drift on large repos and produce a different answer every run. The engine parses the actual files, so two runs on the same commit give the same dictionary, and `data.json` makes every table traceable back to its source file. The model's job is orchestration and a short human summary, not schema invention.

### 5.3 Source 1 — Liquibase changelogs

The engine finds changelogs by content, not just filename, so it is not tied to a directory layout:

- **XML** (`<databaseChangeLog>`): parsed with `xml.etree`. It reads `<createTable>` (table name, `remarks`), nested `<column>` (`name`, `type`, `remarks`, `defaultValue`), and `<constraints>` (`primaryKey`, `nullable`, `foreignKeyName`, `references`). It also applies later `<addColumn>`, `<addPrimaryKey>`, `<addForeignKeyConstraint>`, and `<dropColumn>` change sets so the final table reflects the full changelog, not just its first definition. `<include>`/`<includeAll>` are followed.
- **Formatted SQL** (`--liquibase formatted sql`): handed to the SQL parser (Source 2), with change-set markers stripped.
- **JSON**: same shape as XML, parsed with `json`.

Provenance for each table/column is recorded as `liquibase` with the file and change-set id.

### 5.4 Source 2 — Raw SQL and JDBC

The engine extracts `CREATE TABLE` (and `ALTER TABLE ... ADD COLUMN`, `ADD CONSTRAINT`) from:

- standalone `.sql` files (`schema.sql`, `data.sql`, init scripts, Flyway `V*.sql`), and
- **SQL embedded in Java string literals** — the JDBC case. The engine concatenates adjacent Java string literals (handling `+` concatenation and text blocks `"""..."""`), then scans the result for `CREATE TABLE` statements. This is how it finds `jdbcTemplate.execute("CREATE TABLE ...")`.

A small SQL DDL parser (regex + bracket-aware column splitting) extracts column name, type with size (`VARCHAR(120)`, `NUMERIC(10,2)`), `NOT NULL`, `PRIMARY KEY`, `DEFAULT`, and table-level `PRIMARY KEY (...)` / `FOREIGN KEY (...) REFERENCES ...` clauses. Provenance is `sql` (for `.sql` files) or `jdbc` (for SQL found inside Java).

### 5.5 Source 3 — JPA / Hibernate entities

For each Java file containing `@Entity`, the engine parses annotations to reconstruct the table Hibernate would generate:

- `@Table(name=...)` or, if absent, the class name converted to the Hibernate default (`CamelCase` → `snake_case`).
- Each persistent field → a column: name from `@Column(name=...)` or the field name snake-cased; type mapped from the Java type (`Long`→`BIGINT`, `String`→`VARCHAR(255)`, `Integer`→`INT`, `BigDecimal`→`NUMERIC(19,2)`, `LocalDate`→`DATE`, `LocalDateTime`→`TIMESTAMP`, `boolean`→`BOOLEAN`, enums→`VARCHAR(255)`, etc.), honoring `@Column(length=, nullable=, precision=, scale=)`.
- `@Id` → primary key; `@GeneratedValue` noted as the default.
- `@ManyToOne` / `@JoinColumn` → a foreign key; `@OneToMany`/`@ManyToMany` mapping tables are noted where declared.
- `@Transient` fields are skipped.

Provenance is `hibernate`. The mapping mirrors Hibernate's default naming and type strategy; it is documented as an approximation of generated DDL, not a guarantee.

### 5.6 Merge and relationships

Tables discovered under the same name from different sources are merged; columns union by name, and the table's `source` becomes `mixed` when more than one tool defines it. Relationships are resolved in two passes:

1. **Explicit** — foreign keys declared in Liquibase `<constraints references>`, SQL `FOREIGN KEY`, or JPA `@JoinColumn`/`@ManyToOne`.
2. **Inferred** — any column named `<x>_id` whose `<x>` (singular/plural-normalized) matches a known table name, when no explicit FK already covers it. This is what connects a Hibernate entity to a Liquibase table that has no JPA mapping. Inferred FKs are flagged as inferred in the data so they are visibly distinct from declared ones.

### 5.7 Deterministic per-table facts ("size")

With no live database there are no row counts. "Size" is reported as schema size, all derived deterministically from the parsed definition:

- **column count**,
- **declared column sizes** (varchar length, numeric precision/scale),
- **estimated row width in bytes** — sum of per-column type widths (`BIGINT`=8, `INT`=4, `BOOLEAN`=1, `DATE`=4, `TIMESTAMP`=8, `VARCHAR(n)`≈`n`, `NUMERIC(p,s)`≈`p/2+1`, `TEXT`≈ a fixed estimate), clearly labeled "estimated".

No row counts, table byte sizes, or index statistics are fabricated.

### 5.8 Descriptions

Descriptions are surfaced, not invented:

- **Real remarks/comments** are used verbatim when present — Liquibase `remarks`, SQL `COMMENT ON`, JPA there are none natively.
- When a table or column has no remark, the engine generates a **humanized label** from the identifier (`first_name` → "First name", `owner_id` → "Owner (foreign key)") so the dictionary is readable. These are clearly derived-from-name, never claimed as authoritative documentation.

## 6. The Website

A single self-contained `index.html`, light theme, no build step. Header with project name, generation time, and counts (tables, columns, and a per-source tally). Then three tabs:

### Tab 1 — Tables
A searchable list. Each table is a card showing: name, a source badge (Liquibase / SQL / JDBC / Hibernate / Mixed), origin file, description, column count and estimated row width, and a column table (name, type, size, nullable, PK/FK markers, default, description). Foreign-key columns link to their referenced table.

### Tab 2 — ER Diagram
A hand-drawn, Excalidraw-style SVG: each table is a pastel rounded box listing its key columns; relationships are lines (declared = solid, inferred = dashed) with a small crow's-foot at the many end. The wobble comes from an SVG `feTurbulence`+`feDisplacementMap` filter and the **Caveat** handwriting font, matching this collection's diagram style. Layout is computed in vanilla JS (simple layered placement by relationship depth). Boxes are color-coded by source.

### Tab 3 — Query
An in-browser SQL console. On load it builds an in-memory SQLite database from the engine's normalized DDL (and seeds any `INSERT`s discovered in the source so queries return rows), using `sql.js`. The user types any SQL and sees the result grid, row count, and timing. A few starter queries are offered as clickable chips. Errors are shown plainly. This is read-and-explore over the schema, isolated to the browser tab.

### 6.1 External resources (and why)
The page is self-contained except for two justified resources:
- **sql.js** (SQLite WASM) — there is no pure-vanilla way to execute arbitrary SQL in a browser; writing a SQL engine is out of scope, so the one accepted dependency runs the query tab.
- **Caveat** web font — required for the hand-drawn diagram style used across this collection; falls back to a system cursive if offline.

Everything else (CSS, layout, ER rendering, table rendering) is inline vanilla JS/CSS with no framework.

## 7. The Sample App (`sample/`)

A runnable petshop service that deliberately defines its schema across all three mechanisms so the skill has something real and mixed to discover. It is **only a fixture** — the skill never reads it specially.

- **Stack**: Java 25, Spring Boot 4.0.6, H2 (in-memory), Maven.
- **3 tables via Liquibase** (`db/changelog`): `owner`, `pet`, `visit`.
- **2 tables via raw SQL/JDBC** (`JdbcTemplate.execute` at startup): `product`, `supplier`.
- **2 tables via JPA/Hibernate** (`@Entity`, `ddl-auto=update`): `appointment`, `vaccination`.

Relationships span the sources to exercise the merge and inference:
- `pet.owner_id → owner`, `visit.pet_id → pet` (Liquibase, declared).
- `product.supplier_id → supplier` (SQL/JDBC, declared).
- `appointment.pet_id → pet`, `vaccination.pet_id → pet` (Hibernate columns, **inferred** because `pet` is not a JPA entity).

```
                 +-----------+        +-----------+
   Liquibase --> |  owner    |<-------|   pet     |<----+----- (inferred) appointment   <- Hibernate
                 +-----------+        +-----------+     |
                                          ^             +----- (inferred) vaccination   <- Hibernate
                          (declared) visit +
   SQL/JDBC -->  +-----------+        +-----------+
                 | supplier  |<-------|  product  |
                 +-----------+        +-----------+
```

## 8. Invocation

```
/data-dict                 scan the current repository, write the report, open it
/data-dict sample/         scan a subdirectory (e.g. the sample app)
/data-dict <path>          scan any path
```

Output is written to `data-dict-report/index.html` and `data-dict-report/data.json` in the current working directory, then opened in the browser.

## 10. Suggestions and Improvements

- **Flyway-aware versioning** — order `V*__*.sql` by version and show migration history per table.
- **Drift check** — diff the JPA-derived DDL against the Liquibase/SQL DDL for the same table and flag mismatches (a real source of production bugs).
- **More dialects** — Postgres/MySQL-specific types in the SQLite normalizer for a higher-fidelity query tab.
- **Index & unique-constraint capture** — currently keys and FKs only.
- **Live mode** — optional flag to connect to a running database for real row counts and sizes, kept strictly opt-in.

## 11. Critiques and Limitations

| Limitation | Impact | Mitigation |
|---|---|---|
| Hibernate DDL is reconstructed from annotations, not generated by Hibernate | Edge cases (custom naming strategies, `@SecondaryTable`, inheritance) may differ | Documented as an approximation; covers the common annotation set |
| The query tab uses SQLite, not the project's real database | Vendor-specific SQL won't run identically | Tab is for schema exploration; types normalized, exotic types → `TEXT` |
| No live database means no real sizes/row counts | "Size" is schema size only | Clearly labeled "estimated"; never fabricates row stats |
| Inferred FKs can over- or under-connect | A `status_id` with no `status` table won't link; a coincidental name might | Inferred links are visually distinct (dashed) from declared ones |
| YAML Liquibase is best-effort (no stdlib YAML) | A YAML-only project may under-report | XML/SQL/JSON fully supported; YAML noted as future work |
| sql.js and Caveat load from CDN | Query tab and exact diagram font need network on first load | Both degrade gracefully; the rest of the report is fully offline |

## 12. Success Criteria

- Runs on any repo with `python3` and produces `data-dict-report/index.html` + `data.json` with no third-party Python packages.
- On the sample app it discovers all **7 tables** and labels each with the correct source (3 Liquibase, 2 SQL/JDBC, 2 Hibernate).
- It resolves the declared FKs and infers the two Hibernate→`pet` relationships.
- Tab 1 lists every table with columns, sizes, and source badges; Tab 2 draws the ER diagram with declared vs inferred edges; Tab 3 runs real SQL in the browser against the discovered schema.
- The site is light-theme and self-contained apart from the two documented external resources.
- `install.sh` places the skill under `~/.claude/skills/data-dict`; `uninstall.sh` removes it cleanly.
