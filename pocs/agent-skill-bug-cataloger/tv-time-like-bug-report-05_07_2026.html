01

Bug

  • Refresh Artwork gave library movies wrong covers (e.g. a Ponzi-scheme photo).
  • Titanic showed the ship, Barbie the logo, The Invisible Man the novel, F1 a race photo.
  • server/metadata.ts preferred Wikipedia lead-images over IMDb posters.
  • A plain movie title resolved to the wrong Wikipedia article.
  • The re-scan filter matched 'wikipedia.org' but images were on 'upload.wikimedia.org', so it re-scanned almost nothing.

Solution

  • Made the sync prefer IMDb (media-amazon.com) posters, matched by title + type + year.
  • Changed needsArtwork to re-scan any poster not from media-amazon.com instead of the missed wikipedia.org check.
  • Added gentler batching (chunks of 6) to avoid HTTP 429 rate-limiting that returned zero updates.
  • Verified in the DB: Titanic, Barbie, Nerve, F1, Aladdin, The Invisible Man all resolved to correct IMDb posters.
  • Only unresolvable titles remained on the monogram placeholder; typecheck passed.
02

Bug

  • Searching 'xmen' or 'xman' under the Movies filter returned nothing.
  • The search endpoint returned only TV shows, never movies.
  • server/media.ts only queried a movie source when TMDB_ACCESS_TOKEN was set.
  • With no token the iTunes fallback returned resultCount 0 for every movie query.
  • So movie results were effectively empty in the default configuration.

Solution

  • Extracted a searchMovies path that falls back to the IMDb suggestion endpoint when no TMDB token exists.
  • IMDb reliably returned X-Men and other titles with posters, unlike the dead iTunes call.
  • Merged movie results into searchMedia and de-duplicated against local library items.
  • Verified live: 'X-Men (2000)' now appears with a poster under the Movies filter.
  • Unit tests passed (3 pass, 0 fail).
03

Bug

  • Adding a title from Discover looked like nothing happened.
  • The added item stayed in the grid still showing 'Add to library'.
  • The server add succeeded (HTTP 201, item in library).
  • The bug was client-side: useLibraryActions only invalidated the 'library' query.
  • The Discover/catalog query was never refetched, so the stale grid kept the added item.

Solution

  • Added a refreshDiscover that invalidates the discover/catalog queries on add and remove.
  • Now adding a title refetches Discover so the owned item leaves the grid.
  • Verified with Playwright: adding Love Island removed it from Discover immediately.
  • The embedded-trailer and ESC-to-close flows were confirmed working in the same pass.
04

Bug

  • Titles already in the library reappeared in Discover picks and search results.
  • The same show could show up as both owned and a fresh suggestion.
  • Discover and search did not exclude items the user already owned.
  • Root cause: no shared owned-title key was used to filter candidate lists.

Solution

  • Added libraryTitleKey / libraryTitleKeys helpers in server/db.ts for a normalized owned-title set.
  • Applied an 'available' filter in search (media.ts) and the AI catalog (ai.ts) to drop owned titles.
  • Verified overlap counts dropped to 0 for first picks and for a 'severance' search.
  • Confirmed a removed/added title no longer double-listed across library and Discover.
