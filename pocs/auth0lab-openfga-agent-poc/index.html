Auth0Lab inspired agent authorization

OpenFGA rules before agent context

This proof of concept teaches a local agentic RAG authorization pattern: a human principal delegates to an agent, retrieval finds candidate documents, and relationship checks decide which documents can cross into the answer.

What this is

A small Python teaching project for relationship-based access control in an agent workflow. It does not call Auth0, an OpenFGA server, a vector database, or a language model. Its value is the authorization shape.

ID

Principal

The user is the human identity behind the request. The agent does not get free access just because it can retrieve text.

AG

Delegation

`agent:secprep` must have `delegate` for the user before retrieval is allowed to proceed.

RB

ReBAC

Access comes from relationship tuples: users own folders, view folders, view documents, and documents inherit folder viewers.

AI

RAG boundary

Candidate documents are checked with `can_cite` before they are included in the final text.

Why it matters

          
Agentic systems can retrieve more than a user should see unless authorization is treated as part of the retrieval boundary.

        

        

          

            Least privilege survives retrieval
            
The retriever can find candidates broadly, but `can_cite` narrows the answer to what the principal may use.

          

          

            Delegation is separate
            
A user may have document access and still not allow this agent to act. Dana owns finance content but cannot delegate to `agent:secprep`.

          

          

            Model drift is visible
            
The Python evaluator mirrors `model.fga`; the README calls out that production code should execute a real authorization service.
