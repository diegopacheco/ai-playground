# HTML as Contract — Design Doc

## 1. Summary

A single HTML file acts as a living engineering contract. It is **dual-reader**:
a human opens it in a browser and sees a styled, tabbed page; an agent reads the
same file as semantic markup, extracts the rules, and writes a compliance report
back into it.

Rules (testing, design, code) live in tab 1 with good/bad code pairs. A
compliance report lives in tab 2 and is regenerated by an agent on demand. The
whole thing ships as a Claude Code skill `/html-as-contract` with install /
uninstall scripts for the global Claude install and a `sample/` project to run
against.

## 2. Background

Inspired by Tariq (Claude Code team) and the "HTML as contract" idea: markup is
a strong interface for agents because it is both human-renderable and natively
parseable by an LLM. One artifact carries the spec, the good/bad code, and the
current compliance status — no second format to keep in sync.

- Markdown carries text but not the rendered UX.
- JSON carries structure but is not human-friendly.
- HTML carries both, and an agent can read **and** rewrite regions of it.

## 3. Goals / Non-goals

### Goals
- One self-contained HTML file = rules + good/bad code + compliance report.
- Light, modern theme only. Never dark.
- Tabbed UI: tab 1 = rules, tab 2 = report.
- Rules grouped by category (testing, design, code) with stable machine hooks.
- Report regenerated by an agent, callable by a human or another LLM.
- Ships as a skill with `/html-as-contract` (review/accept/suggest) and
  `/html-as-contract check` (compliance).
- Global install / uninstall scripts plus a `sample/` project.
- Vanilla only — no frameworks, no runtime libraries.

### Non-goals (POC)
- No real AST parser or static-analysis engine; the agent reads code directly.
- No CI / git-hook wiring (left as a future trigger).
- No multi-file or multi-repo aggregation.
- No dark theme.

## 4. Core concept: dual-reader artifact

```
                 contract.html
        +-----------------------------+
        |  chrome: light theme, tabs  |   <- presentation only
        |-----------------------------|
        |  TAB 1  rules region        |   <- human-authored, agent READ-ONLY
        |    [rule cards w/ data-*]   |
        |-----------------------------|
        |  TAB 2  report region       |   <- agent-OWNED, regenerated
        |    [per-rule status]        |
        |-----------------------------|
        |  agent-protocol block       |   <- self-executing instructions
        +-----------------------------+
              ^                    ^
       human (browser)      agent (raw markup)
```

The same file is the spec a human reviews and the input/output an agent works on.

## 5. The contract HTML

Single file. Light modern theme. Plain `<button>` tabs toggling sections via a
few lines of vanilla JS. Three logical regions with explicit ownership.

### 5.1 Region ownership (write boundaries)

| Region              | Author          | Agent may write? | Notes                                  |
|---------------------|-----------------|------------------|----------------------------------------|
| chrome (CSS/tabs)   | template        | no               | presentation only                      |
| rules (tab 1)       | human           | no (read-only)   | source of truth for the contract       |
| report (tab 2)      | agent           | yes (replace)    | regenerated wholesale each `check`      |
| agent-protocol      | template        | no               | the prompt the agent follows           |

The agent replaces **only** the report region. It must never edit rules. The
report region is delimited so regeneration is idempotent and safe:

```
<section id="compliance-report" data-region="agent-owned">
  ...everything here is replaced on each check...
</section>
```

### 5.2 Tab 1 — Rules

Each rule is semantic HTML with stable hooks so the agent can locate and reason
about it unambiguously.

Rule fields:

| Field        | Hook              | Meaning                                             |
|--------------|-------------------|-----------------------------------------------------|
| id           | `data-rule-id`    | stable id, e.g. `code-no-null`                      |
| category     | `data-category`   | `testing` \| `design` \| `code`                     |
| severity     | `data-severity`   | `error` \| `warning` \| `info`                      |
| check type   | `data-check`      | `static` (pattern, deterministic) \| `semantic` (judgment) |
| statement    | text              | the rule itself                                     |
| rationale    | text              | why it matters                                      |
| good / bad   | code blocks       | a passing snippet and a violating snippet           |

Starter rules across the three categories:

- **testing** — a test has only one assert (`static`, warning).
- **design** — a DAO is used by at most one service (`semantic`, error).
- **code** — never return null (`static`, error).

Rule card shape (illustrative, no styling shown):

```
<article class="rule" data-rule-id="code-no-null"
         data-category="code" data-severity="error" data-check="static">
  <h3>Never return null</h3>
  <p class="statement">Methods must not return null; return an empty object instead.</p>
  <p class="rationale">Null returns push NPE risk onto every caller.</p>
  <pre class="good">User find(id) { ... return User.empty(); }</pre>
  <pre class="bad">User find(id) { ... return null; }</pre>
</article>
```

The `data-check` flag is important: it tells the report how much to trust a
finding. `static` rules are pattern-matchable with high confidence; `semantic`
rules depend on the agent's judgment and are reported as such.

### 5.3 Tab 2 — Compliance report

Agent-owned. Regenerated on each `check`. Carries a run timestamp, a summary,
and one entry per rule.

Report fields per rule:

| Field        | Hook                | Meaning                                          |
|--------------|---------------------|--------------------------------------------------|
| rule id      | `data-rule-id`      | links back to the rule in tab 1                  |
| status       | `data-status`       | `pass` \| `fail` \| `unknown`                    |
| count        | text                | number of violations                             |
| violations   | list                | each: `file:line`, snippet, short explanation    |

Run-level summary:

| Field        | Meaning                                  |
|--------------|------------------------------------------|
| generated-at | ISO timestamp of the run                 |
| scope        | path scanned                             |
| totals       | rules passed / failed / unknown          |

Color coding (light theme): green = pass, red = fail, grey = unknown. Each
violation cites `file:line` so a human can jump straight to it.

### 5.4 Agent-protocol block (self-executing)

The file carries its own operating instructions so any agent can pick it up and
know its job. Non-rendered, readable in source:

```
<script type="text/markdown" id="agent-protocol">
  Read all <article class="rule"> elements. For each rule, scan the target
  scope. Decide pass / fail / unknown. Honor data-check: 'static' = pattern
  match, 'semantic' = judgment (state confidence). Replace ONLY
  #compliance-report. Never edit the rules region. Cite file:line per violation.
</script>
```

## 6. The skill

One skill, `html-as-contract`, dispatching on its argument.

| Invocation                   | Mode      | Interactive | Caller         |
|------------------------------|-----------|-------------|----------------|
| `/html-as-contract`          | review    | yes         | human          |
| `/html-as-contract suggest`  | suggest   | yes         | human          |
| `/html-as-contract check`    | compliance| no          | human or LLM   |

### 6.1 Review mode — `/html-as-contract`
- If no `contract.html` exists in the target, scaffold one from the template
  (chrome + empty rules + empty report + agent-protocol).
- Show current rules; let the user accept, edit, or remove.
- Write accepted rules into the rules region (tab 1). Human-in-the-loop.

### 6.2 Suggest mode — `/html-as-contract suggest`
- Agent scans the codebase and proposes candidate rules (with rationale and
  good/bad snippets) for the user to accept or reject.
- Accepted suggestions are written into the rules region. Nothing is added
  without explicit acceptance.

### 6.3 Check mode — `/html-as-contract check`
- Reads rules from tab 1, follows the agent-protocol block.
- Scans the target scope, evaluates each rule per its `data-check` type.
- Regenerates the report region (tab 2) with status, violations, summary, and
  timestamp.
- Non-interactive and idempotent so a human, an agent, or (later) CI can call
  it. Touches only the report region.

## 7. Install / uninstall

Skill installs into the global Claude skills directory.

- `install.sh` — copies `skill/` into `~/.claude/skills/html-as-contract/`.
- `uninstall.sh` — removes `~/.claude/skills/html-as-contract/`.

Both are plain bash, no comments, no emojis, no sleeps. Idempotent: install
overwrites; uninstall is a no-op if the target is absent.

## 8. Sample project

`sample/` is a runnable project with intentional rule violations so a `check`
produces a non-empty report.

- `sample/contract.html` — scaffolded contract with the starter rules.
- `sample/src/...` — small source set seeded with violations: a method returning
  null, a test with multiple asserts, a DAO shared by two services.

Running `/html-as-contract check` inside `sample/` rewrites
`sample/contract.html` tab 2 with the findings.

## 10. Workflows

```
Author:   /html-as-contract          -> review/accept rules  -> tab 1 updated
Propose:  /html-as-contract suggest   -> agent proposes rules -> tab 1 updated
Verify:   /html-as-contract check     -> agent scans scope    -> tab 2 replaced
Human:    open contract.html          -> read tabs in browser
```

## 11. Tech choices / constraints

- Vanilla HTML + CSS + a few lines of JS for tabs. No frameworks, no libraries.
- Light modern theme only. No dark mode, no theme toggle.
- HTML is the single source of truth — no parallel JSON sidecar.
- Skill logic lives in `SKILL.md` as instructions; no separate runtime.

## 12. Decisions and open questions

Decisions taken for the POC:
- Single HTML file, two tabs, region ownership enforced via a delimited
  agent-owned report section.
- Manual trigger via the slash command (no hooks/CI yet).
- Default scope = current project root; `sample/` scans `sample/src`.
- `static` vs `semantic` check tagging so report confidence is honest.

Open for later:
- Trigger on git pre-commit or CI.
- Optional embedded JSON island mirroring rules for stricter machine reads.
- Multi-path / multi-repo aggregation.
- Rule versioning and a changelog of accepted rules.
