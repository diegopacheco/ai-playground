## Problem Statement

Writing Playwright tests by hand is slow, brittle, and the artifact most QA testers produce — plain-English test cases in Jira, Notion, or a Google Doc — never becomes executable code. A QA tester writes "log in with a bad password, see the error toast," and a developer has to translate that into selectors, waits, and assertions. The translation step is where tests die.

The goal: a hosted web playground where anyone pastes a plain-English test case + a URL, watches an LLM drive the live page, and downloads a clean, human-quality Playwright `.spec.ts` file. No install. No account. Shareable run links for virality.

## What Makes This Cool

- **Watching the agent work is the demo.** A streamed view of the LLM clicking through the real page is inherently magnetic — it's the same "watch it think" thrill people had the first time they used Browser Use, but with a tangible artifact at the end.
- **The output is what makes it different.** Stagehand, Skyvern, and Browser Use emit agent-shaped scripts that look like nobody you'd hire wrote them. This emits code with PageObject pattern, named functions, real assertions — the kind of file a senior engineer would commit without rewriting.
- **One-link sharing is the viral mechanic.** Every run produces a permalink that replays the agent's steps and shows the generated script. People share "look what it built" links on Twitter/HN, each one is a free ad.
- **Zero install means zero friction.** You can convince someone to try this in one sentence: "paste your test, watch it work."

## Constraints

- **Implementation path is fixed:** all code lives inside `/Users/diegopacheco/git/diegopacheco/ai-playground/pocs/gstack-poc`. No files written above that directory. Frontend, runner backend, infra config — everything sits under this POC folder.
- Open source, side project, open-ended timeline.
- Hosting costs grow with usage (real browsers + vision LLM tokens). Must have a credible cost-control story before the HN demo, or success becomes a bill.
- Zero-config — no "teach it your repo's style" step at v1. The output must look reasonable for a generic Playwright project.
- Self-healing was picked as the magic-moment demo, but a hosted playground doesn't naturally own a user's repo. The doc proposes a workaround (Open Questions).

## Premises

1. **P1 (agreed)** — committed Playwright scripts in git is the right artifact in 2026. Code wins over ephemeral agent runs for audit, debug, and CI determinism.
2. **P2 (agreed)** — self-healing is a real daily-driver feature, not just demoware. (Tension with C, see Open Questions.)
3. **P3 (revised)** — zero-config is a hard requirement. The tool must produce reasonable Playwright code without first ingesting the user's repo conventions. Implication: the output is "good generic Playwright," not "matches your style." That's fine for v1.
4. **P4 (revised)** — generation speed matters. 30-60 seconds per test of LLM-driving-the-browser is too slow for a fun playground experience. Must mitigate via streaming UX (the user is watching, so wait feels active not idle), parallel execution where possible, and aggressive caching of repeated page DOM snapshots.

## Approaches Considered

### Approach A: CLI tool
Single `qa2pw` binary, no web UI, no daemon. Ships in 1-2 weekends. Generator + `heal` subcommand for repairing drifted tests.

### Approach B: Watcher daemon + PR automation
Long-running process watches a manual-tests directory, auto-converts, auto-runs, auto-opens PRs. Full self-healing loop. ~1 month of weekends. Best daily-driver UX.

### Approach C: Hosted web playground (chosen)
Browser-based: paste English + URL, watch LLM drive the page in real time via streamed CDP/screenshots, download generated `.spec.ts`. Per-run shareable permalink.

## Recommended Approach

**Approach C — Hosted web playground.**

The author's rationale (inferred from the open-source/side-project framing and the choice of viral-demo magic moment): reach matters more than depth for a side project. A CLI ships faster, but a CLI has a hundred competitors and nobody finds it. A playground with a shareable link finds its audience through one viral demo.

### Architecture sketch

- **Frontend:** Next.js (or SvelteKit) app. Left pane: prompt + URL input. Center pane: live streamed browser view (image stream over WebSocket from the runner). Right pane: incrementally building `.spec.ts` with syntax highlighting. Bottom bar: "Download script" + "Share run."
- **Runner backend:** Pool of headless Playwright instances. Each run spawns a fresh browser context. Vision-capable LLM (Claude 4.7 Sonnet, the cheapest model that reliably handles Playwright codegen) drives via the Anthropic SDK with screenshot + DOM-snapshot tool calls.
- **Recording layer:** Every action the LLM takes is logged with the screenshot taken at decision time, the selector tried, whether it succeeded, and the next action. This log becomes both (a) the data for the streamed live view and (b) the source-of-truth for replaying a shared run.
- **Output post-processing:** Raw action log → tree-shaken → emitted as PageObject-pattern Playwright. A small deterministic templater (not LLM) does the final emission so the output structure is consistent.
- **Storage:** Cheap object store (S3/R2) for run replays. Postgres for run metadata. No user accounts at v1 — runs are anonymous + permalinked.
- **Cost control:** Hard timeout per run (90s). Rate limit per IP (5 runs/hour anonymous). Block runs against IP-listed domains (banks, government). LLM token budget per run (~$0.20 ceiling).

### What ships in v1

1. Paste English + URL → watch run → download `.spec.ts`
2. Permalink to replay any run (input + step log + output script)
3. Side-by-side comparison view: "here's how Stagehand/Browser Use would output the same prompt" baked into the demo as a pre-recorded asset
4. Basic abuse protection (timeout, rate limit, domain blocklist)

### What v2 adds

- **Healing flow:** upload your existing `.spec.ts` + URL, the playground re-runs it, identifies failed selectors, proposes patches, you download the patched file. This is how self-healing fits into a hosted playground without owning the user's repo.
- **A CLI version (Approach A)** that's just the same generator engine, packaged for local runs. The web playground becomes the marketing surface; the CLI becomes the daily driver.

## Open Questions

1. **Self-healing in a hosted context.** Picked as the magic-moment demo, but the playground doesn't own a user's repo. Proposed v2 answer: accept uploaded `.spec.ts + url`, return patched file. Confirm whether that's enough to deliver the "magic moment," or whether the magic moment for v1 should be reframed as just "watch it build."
2. **Browser pool: own infra vs Browserbase.** Browserbase removes the operational burden but costs more per run and creates a dependency. Self-hosting is cheaper at low volume but anything past a viral spike will melt. Decision deferred until volume is observable; build the runner abstraction so the backend is swappable.
3. **Model choice and cost.** Claude 4.7 Sonnet with vision is the safest default for quality, but per-run cost may exceed $0.20 for complex flows. Worth piloting Haiku 4.5 (cheaper, faster, may sacrifice script quality) as a separate "draft mode" toggle.
4. **What's the output license/usage story?** Generated scripts go to the user's repo. License clearly on download (MIT-like) so QA teams can use them at work without legal review.

## Success Criteria

- **First milestone:** a stranger pastes a URL + prompt, gets a working `.spec.ts` in under 90 seconds, runs it locally with `npx playwright test`, and it passes on the first try at least 60% of the time on common SaaS login/signup/checkout flows.
- **Second milestone:** a single shared-run link does 1,000+ views on social.
- **Third milestone:** 100+ unique repos on GitHub contain a `tests/` file with the playground's generation-signature comment header.

## Distribution Plan

- **Hosted app** at a memorable domain (qa2pw.dev or similar). Self-explanatory landing page = the playground itself.
- **GitHub repo** open from day one. Even though the product is hosted, the engine is open source — anyone can self-host or fork. This is the credibility move that earns HN trust.
- **Launch surfaces:** HN Show, /r/QualityAssurance, /r/programming, Playwright community Discord. Cross-post the demo video. Hold one big-name Twitter share for launch day.
- **CI/CD:** GitHub Actions builds the container image with podman on tag, pushes to GHCR, deploys to Fly.io or Railway (both have generous free tiers; matters for a cost-bound side project).

## Next Steps

All code goes under `pocs/gstack-poc/`. Suggested layout:
```
pocs/gstack-poc/
  runner/        # standalone Playwright + LLM script (step 1)
  protocol/      # websocket message types shared by runner + web (step 2)
  web/           # Next.js app (step 3)
  infra/         # Containerfile, podman-compose, deploy config (step 5)
```

1. **Spike the runner** — smallest possible standalone script in `pocs/gstack-poc/runner/`: takes English + URL, drives Playwright with Claude vision, emits a `.spec.ts`. No web UI yet. Goal: prove the generator works end-to-end on three real test cases (login, signup, broken-link click).
2. **Add the streamed-view protocol** — WebSocket relay from runner to client, sending base64 screenshots + JSON action events. Build a tiny standalone viewer first; don't wrap a full Next.js app until the protocol works.
3. **Wrap in Next.js** — the actual playground UI. Three panes, one button. Plain Tailwind, no design system needed for v1.
4. **Persistence + permalinks** — store run replays in S3/R2, metadata in Postgres. Permalink route renders the replay viewer.
5. **Cost guardrails** — timeout, rate limit, domain blocklist, token budget. Do this BEFORE public launch, not after.
6. **The demo video** — 60 seconds: paste a prompt, watch the agent work, download the file, run it locally, see the green checkmark. This is the launch asset.
